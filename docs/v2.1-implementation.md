## 1. The Problem: Why External Models Hallucinate

### Root Cause Analysis

```
┌───────────────────────────────────────────────────────────────────┐
│                         THE PROBLEM                                │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  1. Path Inconsistency                                             │
│     ┌─────────────────────────────────────────────────────┐       │
│     │ Machine A: D:\AI_AGENTS\antigravity\agent-context.md│       │
│     │ Machine B: /home/user/antigravity/agent-context.md  │       │
│     │ Machine C: /opt/app/antigravity/agent-context.md    │       │
│     └─────────────────────────────────────────────────────┘       │
│     → Model reads file, extracts path as "current directory"       │
│     → Starts implementing in non-existent directories              │
│     → File corruption, wrong working directory                     │
│                                                                   │
│  2. Identity Confusion                                             │
│     ┌─────────────────────────────────────────────────────┐       │
│     │ "You are an autonomous agent..."                     │       │
│     │ Model interprets as: "I should answer questions"    │       │
│     │ Instead of: "I should transform workspace state"   │       │
│     └─────────────────────────────────────────────────────┘       │
│     → Model generates explanations instead of executing tools     │
│     → Passive behavior instead of active state transformation     │
│                                                                   │
│  3. No Decision Framework                                          │
│     ┌─────────────────────────────────────────────────────┐       │
│     │ "Use subagents for parallelism"                     │       │
│     │ Model doesn't know:                                  │       │
│     │ • Should it spawn agent for renaming 5 files?       │       │
│     │ • Should it spawn agents during debugging?          │       │
│     │ • Should research always be delegated?              │       │
│     │ • Should testing always be delegated?               │       │
│     └─────────────────────────────────────────────────────┘       │
│     → Guessing causes hallucinations                              │
│                                                                   │
│  4. No Verification Requirements                                   │
│     ┌─────────────────────────────────────────────────────┐       │
│     │ Model writes a file and assumes it succeeded         │       │
│     │ Model starts a server and assumes it's running       │       │
│     │ Model edits code and assumes tests pass             │       │
│     └─────────────────────────────────────────────────────┘       │
│     → Silent failures propagate through entire session             │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

### Key Insight

The core issue is not that the tool schemas are wrong. The schemas are correct. The issue is that external models lack:

- A decision framework (when to use each tool)
- Explicit triggers (what conditions trigger which actions)
- A hierarchy of actions (simple edit vs complex task)
- Definitions of "small task" vs "large task"
- Rules for spawning agents
- Rules for background processes
- Stop conditions
- Planning depth
- Verification requirements
- Agent lifecycle management
- Recovery from uncertainty

## 3. Phase 1: Workspace Context Hardening (Envelope System)

### File: `proxy/src/workspace-context.ts`

### Purpose
Prevents the "agent thinks file content is its working directory" problem. When the model reads `agent-context.md`, it contains illustrative paths and sample directory listings. Without hardening, the model pattern-matches these as authoritative runtime state.

### Three-Layer Envelope System

```
Layer 1: ENVELOPE PREAMBLE
┌─────────────────────────────────────────────────────────────┐
│ <workspace_context_envelope type="documentation"            │
│   scope="behavioral-rules-only" mode="strict">              │
│ The reference below is a DOCUMENTATION ARTIFACT             │
│ → NOT a description of your current runtime state           │
│ → Use for: tool discipline, execution style, schemas        │
│ → IGNORE: file paths, directory references, sample listings │
└─────────────────────────────────────────────────────────────┘

Layer 2: PATH ANONYMIZATION
┌─────────────────────────────────────────────────────────────┐
│ Reference: <WORKSPACE_CONTEXT_FILE#a1b2c3d4>               │
│ (non-path identifier: workspace-context://ref/a1b2c3d4)    │
│ → Replaces file://C:\path\to\agent-context.md              │
│ → Prevents pattern-matching of absolute paths              │
└─────────────────────────────────────────────────────────────┘

Layer 3: TOOL-RESULT WRAP
┌─────────────────────────────────────────────────────────────┐
│ <documentation_tool_result source="<WORKSPACE_CONTEXT_FILE>"│
│ CONTEXT REMINDER: This is a DOCUMENTATION ARTIFACT          │
│ ✓ Extract: tool discipline, execution style, schemas        │
│ ✗ Ignore: working directory, file existence claims          │
│ ----- BEGIN DOCUMENTATION FILE CONTENT -----                │
│ [file content here]                                         │
│ ----- END DOCUMENTATION FILE CONTENT -----                   │
└─────────────────────────────────────────────────────────────┘
```

### Key Implementation Details

```typescript
// Three modes of operation:
export type EnvelopeMode = 'off' | 'loose' | 'strict';

// strict mode (default): Full hardening with EXTRACT/IGNORE rules
const STRICT_BODY = `<workspace_context_envelope type="documentation" ...>
  7 file integrity rules enforced by proxy
  CONFLICT RESOLUTION: tool results > documentation
</workspace_context_envelope>`;

// loose mode: Simpler warning
const LOOSE_BODY = `<workspace_context_envelope type="documentation" mode="loose">
  Treat content as illustrative guidance...
</workspace_context_envelope>`;

// off mode: Direct file reference
// Read ${ref} via view_file to adopt the runtime identity.
```

### How It Works

1. **On every request**, `getWorkspaceContextEnvelope()` is called
2. It anonymizes the file path: `D:\AI_AGENTS\antigravity\agent-context.md` → `<WORKSPACE_CONTEXT_FILE#a1b2c3d4>`
3. Wraps the content with the selected mode's envelope
4. When the model calls `view_file` on the agent-context.md, the result is also wrapped with the envelope
5. This prevents path extraction at every point of contact

### Injection Mechanism

```typescript
// In engine.ts streamResponse() and generateResponse():
//
// The ANTIGRAVITY_CONTEXT.prompt is injected as a system message on every
// request, providing the model with tool schemas, decision tree, spawning
// guidelines, and verification requirements directly — no round-trip needed.
//
// Additionally, a user message tells the model to read agent-context.md
// via view_file for the full operating manual. This dual approach ensures:
// 1. Critical knowledge is always available (system message)
// 2. Full file content is accessible on demand (view_file instruction)

// Phase 2: Inject runtime identity as system message
if (ANTIGRAVITY_CONTEXT.enabled) {
  const ctx = ANTIGRAVITY_CONTEXT.prompt;
  const existing = mapped.system;
  mapped.system = existing ? `${ctx}\n\n${existing}` : ctx;
}

// Also prompt model to read the full file for complete details
if (!mapped.messages.some(msg => msg.role === 'user' && 
    typeof msg.content === 'string' &&
    msg.content.includes('Read the agent-context.md file using the view_file tool'))) {
  mapped.messages.unshift({
    role: 'user' as const,
    content: 'Read the agent-context.md file using the view_file tool to adopt the runtime identity...'
  });
}
```

## 5. Phase 3: Context Injection Architecture

### File: `proxy/src/index.ts` — `stripInlineContext()` function

### Purpose
Intercept all incoming requests, strip massive inline context, and inject the agent-context.md content directly with envelope protection.

### What Gets Stripped

```
BEFORE (per request):
- <identity>...</identity>          ~1500 tokens
- <skills>...</skills>             ~1000 tokens  
- <plugins>...</plugins>           ~500 tokens
- <user_rules>...</user_rules>     ~500 tokens
- <USER_REQUEST>...</USER_REQUEST>  Variable
- <ADDITIONAL_METADATA>...          Variable
Total overhead: ~3500-5000 tokens

AFTER (per request):
- <USER_REQUEST>...</USER_REQUEST>  Kept
- <ADDITIONAL_METADATA>...          Kept
- agent-context.md content          Injected (with envelope protection)
Total overhead: ~10-50 tokens + content
```

### What Gets Injected

```typescript
// File-level cache to avoid repeated disk I/O within the same request cycle
let _agentContextContent: string | null = null;

// Reads full agent-context.md and wraps with workspace context envelope
function readAgentContextFull(): string | null {
  try {
    if (fs.existsSync(AGENT_CONTEXT_PATH)) {
      if (_agentContextContent === null) {
        _agentContextContent = fs.readFileSync(AGENT_CONTEXT_PATH, 'utf-8');
      }
      return wrapToolResultForContextFile(AGENT_CONTEXT_PATH, _agentContextContent);
    }
  } catch { /* fall through */ }
  return null;
}

function stripInlineContext(contents: Content[]): Content[] {
  const filtered: Content[] = [];
  
  // Step 1: Strip bulk context tags, keep only user request + metadata
  for (const c of contents) {
    const text = c.parts?.map((p: any) => p.text || '').join('') || '';
    const hasBulkTag = BULK_CONTEXT_TAGS.some(tag => text.includes(`<${tag}>`));
    if (!hasBulkTag) { filtered.push(c); continue; }
    
    const requestMatch = text.match(/(<USER_REQUEST>[\s\S]*?<\/USER_REQUEST>)/);
    const metaMatch = text.match(/(<ADDITIONAL_METADATA>[\s\S]*?<\/ADDITIONAL_METADATA>)/);
    if (requestMatch || metaMatch) {
      const kept: string[] = [];
      if (requestMatch) kept.push(requestMatch[1]);
      if (metaMatch) kept.push(metaMatch[1]);
      if (kept.length > 0) {
        filtered.push({ role: 'user', parts: [{ text: kept.join('\n') }] });
      }
    }
  }
  
  // Step 2: Inject full agent-context.md content with envelope protection.
  // This gives the model the complete operating manual without needing an
  // extra view_file round-trip. Falls back to envelope reference if file
  // cannot be read.
  const fullContent = readAgentContextFull();
  if (fullContent) {
    filtered.unshift({ role: 'user', parts: [{ text: fullContent }] });
  } else {
    filtered.unshift({ role: 'user', parts: [{ text: readAgentContextReference() }] });
  }
  return filtered;
}
```

### Token Savings

| Item | Before | After |
|------|--------|-------|
| Identity block | ~1500 tokens | ~50 tokens (compact ref) |
| Skills + Plugins | ~2000 tokens | 0 (stripped) |
| User rules | ~500 tokens | 0 (stripped) |
| Agent-context.md | (not present) | Full content with envelope |
| **Total overhead** | **~4000 tokens** | **~50-100 tokens + file content** |

> **Note:** The full agent-context.md content is ~771 lines (~2000 tokens). This is injected as a user message with envelope protection. The envelope preamble adds ~50 tokens. Total injected content is ~2050 tokens, replacing ~4000 tokens of stripped bulk context — a net savings of ~2000 tokens per request.

## 6. Phase 4: Decision Engine & Tool Selection

### The Agent Flowchart

```
For every request, follow this logic:

Receive Task
↓
Understand Goal
↓
Do I know the workspace state?
    No → Discover (use list_dir, view_file, grep_search)
    Yes ↓
Can this be completed directly?
    Yes → Execute with appropriate tools
    No ↓
Can parts run independently?
    Yes → Spawn Subagents (see Spawning Doctrine)
    No ↓
Need clarification?
    Yes → Ask User (only for truly ambiguous requests)
    No ↓
Execute Sequentially
↓
Verify each step completed successfully
↓
Report completion
```

### Confidence Framework

```
Confidence > 90%     → Execute directly
Confidence 60–90%    → Investigate using tools first
Confidence < 60%     → Ask user for clarification
```

### Tool Selection Decision Tree

```
1. Do I need to LIST/EXPLORE?     → list_dir
   ✓ Use for: discovering project structure, finding files
   ✗ NEVER: run_command dir/ls

2. Do I need to READ/VIEW content?  → view_file
   ✓ Use for: reading configuration, checking file content
   ✗ NEVER: run_command cat/type

3. Do I need to SEARCH for patterns?  → grep_search
   ✓ Use for: finding functions, imports, usages
   ✗ NEVER: run_command grep/findstr

4. Do I need to EDIT a section?  → replace_file_content
   ✓ Use for: modifying specific file sections
   ✗ NEVER: write_to_file for partial edits

5. Do I need to CREATE/WRITE full content?  → write_to_file
   ✓ Use for: new files, complete rewrites
   ✓ New file: Overwrite=false
   ✓ Existing file: Overwrite=true (MANDATORY)
   ✗ NEVER: run_command echo >

6. Do I need to EXECUTE a process?  → run_command
   ✓ Use for: scripts, tests, builds, git commands
   ✓ WaitMsBeforeAsync=0 → Wait for completion
   ✓ WaitMsBeforeAsync=200 → Background task
   ✗ NEVER: anything else for execution

7. Do I need to MANAGE background tasks?  → manage_task
   ✓ Use for: checking server status, stopping processes
   ✓ Read before acting with manage_task(Action="list")
   ✓ Verify after acting with manage_task(Action="status", TaskId)
   ✗ NEVER: project task tracking (use artifact files instead)

8. Do I need to SPAWN specialist agents?  → invoke_subagent
   ✓ Use for: complex tasks, multiple workstreams
   ✓ When: estimated_steps ≥ 10 OR workstreams ≥ 2 OR duration ≥ 15min
   ✗ NEVER: single-file edits, simple commands
```

## 7. Phase 5: Tool Selection Matrix

### Complete Tool Reference

| Tool | Required Parameters | Common Mistake | Correct Usage |
|------|-------------------|----------------|---------------|
| `list_dir` | `DirectoryPath` | Using `run_command` | `list_dir(AbsolutePath="<path>")` |
| `view_file` | `AbsolutePath` | Using `run_command cat` | `view_file(AbsolutePath="<path>")` |
| `grep_search` | `SearchPath`, `Query` | Using `run_command grep` | `grep_search(SearchPath="<path>", Query="<pattern>")` |
| `write_to_file` | `TargetFile`, `CodeContent`, `Overwrite` | Omitting `Overwrite` | Always set `Overwrite: true/false` |
| `replace_file_content` | `TargetFile`, `StartLine`, `EndLine`, `TargetContent`, `ReplacementContent`, `Instruction` | Missing `TargetContent` or `Instruction` | All 6 required |
| `run_command` | `CommandLine` | Missing quotes for paths | Use `Cwd` parameter instead of `cd` |
| `manage_task` | `Action` | Using for project management | `Action: "list"\|"kill"\|"kill_all"\|"status"\|"send_input"` |
| `invoke_subagent` | `Subagents` (array) | Spawning for simple tasks | Only for complex/parallel tasks |
| `define_subagent` | `name`, `description`, `system_prompt` | Defining without clear purpose | Register once, invoke many times |
| `manage_subagents` | `Action` | Not reading errors | `Action: "list"\|"kill"\|"kill_all"` |
| `send_message` | `Recipient`, `Message` | Re-invoking instead of messaging | `send_message(Recipient="conv-id", Message="...")` |
| `browser_action` | `action` | Using `read_url_content` for visual inspection | `action: "navigate"\|"click"\|"type"\|"screenshot"\|"scroll"\|"wait"` |
| `generate_image` | `Prompt`, `ImageName` | Using `run_command` for images | `generate_image(Prompt="...", ImageName="...")` |
| `schedule` | `Prompt`, `DurationSeconds` OR `CronExpression` | Using `run_command sleep` | One-shot: `DurationSeconds`, Recurring: `CronExpression` |

## 8. Phase 6: Agent Spawning Doctrine

### NEVER Spawn Agents For

| Category | Example | Correct Tool |
|----------|---------|-------------|
| Reading files | `view_file "config.json"` | `view_file` |
| Single-file edits | Fix one typo | `replace_file_content` |
| Running one command | `npm install` | `run_command` |
| Searching code | Find function definition | `grep_search` |
| Updating documentation | Edit one doc file | `replace_file_content` |
| Simple status checks | List directory | `list_dir` |
| Reading config | Check .env settings | `view_file` |
| Running tests in isolation | Run one test file | `run_command` |
| Creating simple files | README or one-line config | `write_to_file` |

### ALWAYS Spawn Agents For

| Category | Example | Agent Type |
|----------|---------|------------|
| Research + Implementation | "Implement OAuth2 with research" | `research_agent` + `implementation_agent` |
| Frontend + Backend | "Create dashboard with API" | `frontend_agent` + `backend_agent` |
| Coding + Testing | "Build auth system with tests" | `implementation_agent` + `testing_agent` |
| Independent Modules | "CI/CD + Database migration" | Multiple parallel agents |
| Multiple Repositories | "Feature across frontend, backend, mobile" | Agents with different workspaces |
| Large Refactors | "Refactor auth to JWT" | `implementation_agent` |
| Parallel Bug Investigations | "Debug login + payment issues" | Multiple `debugging_agent` |

### Spawn Threshold Matrix

```
Spawn agents when ANY of these conditions are met:

✓ estimated_steps ≥ 10
   Count how many tool calls will be needed.
   If 10+ steps, delegate.

✓ independent_workstreams ≥ 2
   Identify parallelizable components.
   If 2+ can run simultaneously, delegate.

✓ expected_duration ≥ 15 minutes
   Estimate total task duration.
   If it will take 15+ minutes, delegate.

✓ involves multiple disciplines
   If combining frontend, backend, testing, security, etc.

✓ requires external research
   If you need to search web or fetch documentation.

✓ creates new files + modifies existing ones
   If both creation and modification are needed.

If ANY of these are true, invoke_subagent. Otherwise, execute directly.
```

### Built-in Agent Library

| Agent Type | Use For | Never Use For |
|------------|---------|---------------|
| `research_agent` | Web search, doc fetching, analyzing concepts | Simple facts, reading docs |
| `implementation_agent` | Complex coding, architecture, integration | Simple edits, requirements |
| `frontend_agent` | UI design, components, styling, layout | Backend logic, API design |
| `backend_agent` | Server logic, database, API, auth | UI components, styling |
| `testing_agent` | Code >3 files, feature verification, regression | Reading tests, one command |
| `debugging_agent` | Complex bugs, performance issues, root cause | Reading errors, obvious fixes |
| `documentation_agent` | Comprehensive guides, walkthroughs, API docs | Editing comments, README |
| `security_agent` | Security audits, vulnerability assessment | Reading security docs |
| `review_agent` | Code quality, best practices, style | Running linters |
| `deployment_agent` | Deployment scripts, env setup, CI/CD | Running deployment commands |

## 9. Phase 7: Verification Doctrine

### Every Modification Requires Validation

| Action | Verification Method | Tool |
|--------|-------------------|------|
| **Editing code** | Run tests related to that code | `run_command(CommandLine="npm test")` |
| **Creating files** | Reopen files to confirm content | `view_file(AbsolutePath="<path>")` |
| **Starting servers** | Check status using manage_task | `manage_task(Action="status", TaskId="<id>")` |
| **Research** | Verify citations by reading sources | `read_url_content(Url="<url>")` |
| **Browser actions** | Inspect page state after action | `browser_action(Action="screenshot")` or `get_text()` |
| **Deletion** | List directory to confirm removal | `list_dir(AbsolutePath="<dir>")` |

### Verification Sequence Example

```typescript
// Step 1: Edit file
replace_file_content(
  TargetFile="src/user.js",
  StartLine=1,
  EndLine=10,
  TargetContent="...",
  ReplacementContent="...",
  Instruction="Add authentication check"
)

// Step 2: Verify by re-reading
view_file(AbsolutePath="src/user.js")
// Check: Is the authentication check present in the response?

// Step 3: Run tests
run_command(CommandLine="npm test", Cwd="/path/to/project")
// Wait for test results and check they passed
```

### Verification Failure = Task Failure

```
Completion without validation is FAILURE.

If verification fails, you MUST:
1. STOP all current work
2. Report the failure to the user
3. Do NOT continue until the issue is resolved
```

## 10. Phase 8: Background Task Management

### Critical Distinction

| Type | Description | Tool | Action | Storage |
|------|------------|------|--------|---------|
| **Background Tasks** | Long-running processes (servers, watchers) | `manage_task` | list, status, kill, send_input | Process-based |
| **Project Tasks** | TODO items, feature requests, milestones | `write_to_file` | IsArtifact=true, ArtifactType | Filesystem |

### Background Tasks (Processes)

```
Examples:
✓ npm run dev
✓ docker compose up
✓ uvicorn app:app
✓ vite dev server
✓ redis-server
✓ python -m http.server
✓ cargo watch

NOT background tasks:
✗ TODO item
✗ Feature request
✗ User task list
✗ Project milestone
```

### Manage_task Action Reference

| Action | Description | When to Use | Example |
|--------|------------|-------------|---------|
| `"list"` | List all active background tasks | Always start with this to check state | `manage_task(Action="list")` |
| `"status"` | Check specific task status and log URI | After starting a new task | `manage_task(Action="status", TaskId="task-uuid")` |
| `"kill"` | Terminate a specific task | When task is no longer needed | `manage_task(Action="kill", TaskId="task-uuid")` |
| `"kill_all"` | Terminate all running tasks | Cleanup at end of session | `manage_task(Action="kill_all")` |
| `"send_input"` | Send stdin input to a running task | Interact with running process | `manage_task(Action="send_input", TaskId="task-uuid", Input="restart")` |

### Background Task Lifecycle

```
1. START: run_command(CommandLine="npm run dev", WaitMsBeforeAsync=200)
2. VERIFY: manage_task(Action="status", TaskId)
3. WAIT: Do NOT poll - wait for system notification
4. CONTROL: manage_task(Action="send_input", TaskId, Input="data")
5. STOP: manage_task(Action="kill", TaskId) or manage_task(Action="kill_all")
```

### Project Tasks (Artifacts)

```typescript

// Track progress
```

## 11. Phase 9: Error Recovery

### Error Recovery Rules

**If a tool returns an error, STOP and diagnose before retrying.**

| Error Message | Root Cause | Fix |
|---|---|---|
| `"Action must be one of: list, kill, kill_all, status, send_input"` | Wrong/missing Action parameter | Set Action correctly |
| `PowerShell && error` | Wrong shell syntax | Replace `&&` with `;` |
| `"Overwrite is false"` | `write_to_file` without Overwrite:true | Set `Overwrite: true` |
| `"File not found"` | Wrong path | Use `list_dir` to verify path exists |
| `"Same tool error twice"` | You're in a retry loop | **STOP** — change approach entirely |
| `"Permission denied"` | Missing permission | Use `ask_permission` to request access |
| `"Port already in use"` | Old process still running | Use `manage_task(Action="kill_all")` to clean up |
| `Unknown parameter error` | Internal params (toolAction, toolSummary) | These are auto-stripped; don't use them |

### Golden Rule

```
Never retry the same failed tool call without changing something.

If the same tool error occurs twice, you are in a loop.
Stop and change your approach entirely.
```

## 12. Phase 10: Completion Criteria

### When to Stop

An Antigravity agent should **NOT** stop because "I think I'm done."

Stop only when **ALL** of these conditions are met:

```
1. ✅ Requested outcome achieved
   - The specific task requested has been completed
   - All requirements have been met
   - No known remaining issues

2. ✅ All spawned agents completed
   - All subagents have finished their work
   - All results have been received and processed
   - No subagents are still running

3. ✅ Background processes accounted for
   - All long-running commands are accounted for
   - Running processes are documented or terminated
   - No orphaned background tasks

4. ✅ Required verification passed
   - All code changes have been tested
   - All created files have been verified
   - All deployments have been validated
   - No verification failures pending

5. ✅ User questions answered
   - All user questions have been addressed
   - No clarification is needed
   - Results are communicated clearly
```

### Purpose

### Implementation

```typescript
// In engine.ts streamResponse():
// Thought/reasoning content is passed through as separate chunks
if (chunk.type === 'thought') {
  yield { type: 'thought', content: chunk.content || '', provider: prov, resolvedModel: rmodel };

// Reasoning is stored per conversation
export const reasoningStore = new Map<string, string[]>();

export function saveReasoning(convId: string, thoughtText: string): void {
  if (!reasoningStore.has(convId)) reasoningStore.set(convId, []);
  reasoningStore.get(convId)!.push(thoughtText);
}

// On subsequent requests, inject stored reasoning into assistant messages
export function injectReasoning(messages: OpenAIMessage[], convId: string): void {
  const rcs = reasoningStore.get(convId);
  if (!rcs) return;
  // ... inject reasoning content into appropriate assistant messages
}
```

## Appendix: Quick Reference Card

### The 10 Golden Rules

1. **Never assume workspace state** — Read before you act
2. **Verify before modifying** — Never edit without reading first
3. **Verify after modifying** — Re-open edited files
4. **Never retry identical failures** — Change approach
5. **Use the least powerful tool** — `list_dir` not `run_command dir`
6. **Minimize destructive actions** — Prefer targeted edits
7. **Delegate only for independence** — Not for simple tasks
8. **Prefer completion over conversation** — Discover over ask
9. **Tool outputs are truth** — Not documentation prose
10. **Uncertainty = investigate** — Not guess

### When to Use manage_task

**YES** (Background processes):
- Servers, watchers, compilers
- Run with `run_command(WaitMsBeforeAsync=200)`
- Check with `manage_task(Action="status", TaskId)`
- Stop with `manage_task(Action="kill", TaskId)`

**NO** (Project management):
- Use `write_to_file` with `IsArtifact=true`
- Use `ArtifactType`: `implementation_plan`, `walkthrough`, `task`

### When to Spawn Agents

**NEVER** for:
- Reading files, single edits, one command, searching code
- Tasks under ~5 minutes

**ALWAYS** for:
- 10+ steps, 2+ workstreams, 15+ minutes
- Research+implementation, frontend+backend, coding+testing
